# Ответ на претензию о несоответствии FireNotify тендерной документации

Ниже приведены меры, принятые для приведения информационной системы FireNotify в соответствие с обязательными требованиями ТД.

---

## 1. Передача обращений по зоне ответственности

**Требование ТД:** передачи обращений в работу по зоне ответственности; распределение заявок по структурным подразделениям; маршрутизация обращений; назначение ответственных по организационной принадлежности.

**Реализовано:**

- **Структурное подразделение (отделение):** у каждой заявки задаётся поле «Структурное подразделение» (отделение). Список настраивается (КОТЭС, КТЭС, ЖТЭС и др.). Фильтрация списка заявок по подразделению.
- **Назначение ответственного:** администратор назначает на заявку ответственного пользователя (зона ответственности). В карточке обращения отображаются «Ответственный (зона ответственности)» и возможность его смены (только для роли Администратор).
- **Маршрутизация:** заявки распределяются по подразделениям и по назначенным ответственным; список заявок можно фильтровать по подразделению и по ответственному.

---

## 2. Поиск заявок по расширенным параметрам

**Требование ТД:** поиск по автору, местонахождению объекта, структуре, типу проблемы, статусу; комбинированный расширенный поиск.

**Реализовано:**

- **По автору обращения:** в заявке сохраняется автор (пользователь, создавший заявку). В списке заявок — фильтр «Автор» и колонка «Автор».
- **По местонахождению объекта:** поле «Адрес / местонахождение» и отдельный фильтр «Местонахождение объекта» в списке. Общий поиск по тексту (источник, адрес, описание) сохранён.
- **По структуре:** фильтр «Структурное подразделение» (отделение).
- **По типу проблемы:** в заявку добавлено поле «Тип проблемы» (классификация). В списке — фильтр и колонка «Тип проблемы». Справочник типов: Пожарный сигнал, Срабатывание АПС, Ложное срабатывание, Техническая неисправность, Эвакуация, Другое.
- **По статусу:** фильтр по статусу (Новое, В обработке, Проверено, Ложное, Закрыто).
- **По периоду:** фильтры «С» и «По» по дате создания.
- Все перечисленные фильтры работают совместно (комбинированный расширенный поиск).

---

## 3. Редактирование заявки и комментарии

**Требование ТД:** возможность редактирования существующей заявки; добавление комментариев и обновление данных; журнал изменений; диалоговое взаимодействие по заявке; история переписки.

**Реализовано:**

- **Редактирование карточки обращения:** в карточке заявки доступно редактирование полей: источник, адрес/местонахождение, статус, структурное подразделение, тип проблемы, назначенный ответственный (для администратора), описание, общее поле комментариев. Право на редактирование — у ролей Администратор и Оператор.
- **Журнал изменений:** по каждой заявке ведётся журнал изменений (кто, когда, какое действие и при необходимости детали). Отображается на вкладке «Журнал изменений» в карточке заявки.
- **Диалог по заявке / история переписки:** реализованы комментарии к заявке — цепочка комментариев с указанием автора и времени. Вкладка «Комментарии» в карточке заявки; возможность добавлять новый комментарий (роли Администратор и Оператор).

---

## 4. Разграничение прав доступа

**Требование ТД:** разграничение прав доступа к информации; ролевая модель с гибкой настройкой прав.

**Реализовано:**

- В системе реализована ролевая модель с тремя ролями: **Администратор**, **Оператор**, **Наблюдатель**.
- **Администратор:** полный доступ — управление пользователями, назначение ответственных по заявкам, все операции с заявками и сообщениями, отчёты, логи, настройки.
- **Оператор:** создание и редактирование заявок, отправка сообщений администратору, просмотр заявок и комментариев; без управления пользователями и без назначения ответственных.
- **Наблюдатель:** только просмотр заявок, карточек, комментариев и журнала изменений; без создания и редактирования заявок, без отправки заявок администратору. В интерфейсе роль отображается явно: «Наблюдатель (только просмотр)», пункты меню на создание/редактирование для данной роли скрыты.
- Разграничение проверяется на сервере (API) и в интерфейсе (доступные действия и пункты меню в зависимости от роли).

---

## 5. Привилегия «Наблюдатель»

**Требование ТД:** возможность предоставления привилегии «Наблюдатель» для отслеживания всех заявок.

**Реализовано:**

- Роль **«Наблюдатель» (OBSERVER)** реализована и отображается в интерфейсе как «Наблюдатель (только просмотр)».
- Пользователь с этой ролью может входить в систему и просматривать все заявки, карточки обращений, комментарии и журнал изменений.
- Создание и редактирование заявок, отправка заявок администратору, управление пользователями и прочие изменяющие действия для роли Наблюдатель недоступны (скрыты в меню и запрещены на API).

---

## 6. Создание заявок через e-mail

**Требование ТД:** возможность создания заявок через e-mail; SMTP/IMAP интеграция; автоматическое преобразование входящего письма в заявку.

**Реализовано:**

- Реализована **интеграция по протоколу IMAP**: подключение к почтовому ящику, чтение непрочитанных писем и создание по каждому такому письму заявки в системе.
- Поля созданной заявки: источник — «Email»; адрес/местонахождение — тема письма; описание — отправитель и текст письма.
- Настройка через переменные окружения в `backend/.env`: IMAP_HOST, IMAP_PORT, IMAP_USER, IMAP_PASSWORD, IMAP_MAILBOX.
- Запуск вручную: `npm run email:ingest` (из папки backend). Для автоматического приёма — запуск по расписанию (например, cron каждые 5 минут).
- Подробное описание настройки и запуска — в документе `docs/EMAIL_ЗАЯВКИ.md`.

## 7. Класс системы (ITSM / Service Desk)

Система FireNotify после доработок поддерживает:

- приём и учёт обращений (заявок) из интерфейса и по e-mail;
- распределение по зонам ответственности и структурным подразделениям;
- расширенный поиск и фильтрацию;
- редактирование карточек, комментарии и журнал изменений;
- ролевую модель с разграничением прав и привилегией «Наблюдатель».

Указанный функционал соответствует требованиям ТД к системе обработки обращений и учёту заявок.

## Итог

Перечисленные пункты претензии учтены; в системе реализованы механизмы, обеспечивающие соответствие обязательным требованиям тендерной документации по передаче обращений по зоне ответственности, расширенному поиску, редактированию и комментариям, разграничению прав, роли «Наблюдатель» и созданию заявок через e-mail.
